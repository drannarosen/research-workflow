# Page anatomy — the docs-as-knowledge-ecosystem rules

`docs-writing-voice` governs *how a sentence sounds*; this page governs *what a page contains and how it
connects*. For Anna's research software (progenax, stellax, …) the docs site is the project's **single
source of truth and onboarding path** — a living knowledge base, not API output. Two readers must both
be served on the way through: **a new graduate student** (needs motivation, prerequisites, a path) and
**future-you / a collaborator** (needs the theory, the decision, the reference, the test that backs it).

The whole site is a **web, not a stack**: theory ↔ the API that implements it ↔ the validation that
checks it ↔ the reference that grounds it. Every page should make at least one of those links explicit.
Distilled from the live progenax / stellax / papers sites.

## Universal page rules (every page)

- **Frontmatter:** `title` + `description` are the house minimum (enforced by research-workflow's
  `myst-docs-hygiene` hook). Add `subtitle` / `options: {status: draft|stable, last_audited: YYYY-MM-DD}`
  where the page's maturity matters (stellax marks drafts prominently).
- **Orient in one sentence:** open by telling the reader where they are and what they'll be able to do —
  not a cold definition (that's the voice rule; see the SKILL).
- **Link outward, deliberately:** at least one cross-reference to the related theory / API / validation /
  reference page. A page with zero outbound links is a dead end in the knowledge web.
- **Anchor numbers:** real CGS/solar values and a "for the Sun: …" sanity check; label equations
  (`:label:`) so other pages can `{eq}` them.
- **Honesty markers:** mark inference vs. citation; never let "it ran / it's elegant" stand in for "it's
  correct."

## Page types and what each must contain

- **Site / section landing (`index.md`):** what this project/section *is*, who it's for, and a routed
  path in ("New here? start at …"; "Already know X? skip to …"). A `{grid}` of cards to the main
  sections works well. No deep content — it's a switchboard.
- **Getting-started / onboarding:** prerequisites stated up front; install/first-run that actually works;
  ends by handing off to the first theory or how-to page. This is the page a new student opens first —
  optimize it for *not getting stuck*.
- **Theory / derivation:** motivate → frame as a physical question → derive from first principles with
  labelled equations; cite sources with `{cite:p}`/`{cite:t}`; bound any analogy; close with a "what we
  just established" bridge **and a link to the API that implements it**. Depth is the point here.
- **API reference:** auto-generated by the idempotent script (NumPy docstrings, one page per module,
  source links) — see myst-projects-and-workflows. Prose stays terse and scannable; don't force a
  narrative hook onto a symbol table. Link each major symbol back to the theory page it realizes.
- **How-to / recipe:** task-first, copy-pasteable, one goal per page, expected output shown. Assumes the
  concepts (link to theory) rather than re-teaching them.
- **Validation / verification:** the `Property | Tolerance | Measured | Anchor` `{list-table}` (anchor =
  the test path), validation-script figures with quantitative captions, and an explicit pass/fail. This
  is where the docs earn trust — see myst-projects-and-workflows for the table schema.
- **Development / verification log:** **dated** (`YYYY-MM-DD`) entries, newest first, each linking the
  PR/issue and showing the metric/diff that justified the change. This is the project's memory — write it
  for the person (you) trying to reconstruct *why* in a year.
- **Bibliography / references:** one centralized `references.bib`; author-year keys (`Plummer1911`); DOI
  where available; a rendered `{bibliography}` page. For load-bearing sources, a short per-paper note
  page (progenax pattern) so a student learns *why this reference matters*, not just that it exists.

## Anti-patterns (the ecosystem rules these protect)

- A page that documents *what the code does* but never *why* (no theory link, no reference) → it ages
  into a stale restatement of the source.
- Theory with no link to its implementation, or an API page with no link to its theory → the web breaks
  into disconnected silos and students can't trace concept → code.
- A quantitative claim with no validation anchor → exactly the "confidently wrong" failure the whole
  stack exists to prevent.
- A "done"/"validated" status on a page whose linked test doesn't exist yet → status theater.
- Onboarding that assumes the prerequisites it should be stating.

## Hand-offs

- Prose voice / pedagogy moves → the `docs-writing-voice` SKILL (this reference's parent).
- Project config, toc, file-naming, the validation-table *schema* → `myst-expert` →
  `myst-projects-and-workflows.md`.
- Edit-time enforcement of frontmatter + no-legacy-syntax → research-workflow's `myst-docs-hygiene` hook.
- Tying doc claims to tests as an evidence discipline → research-workflow `reference-parity-audit` /
  `provenance-of-constants`.
