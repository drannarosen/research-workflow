A change is not done when the code works — it is done when nothing it touched still lies. Every rename, signature change, units/default change, or behavioral shift silently falsifies the prose around it: a docstring describing the old return, a comment citing the old formula, a README claiming the old behavior, a committed figure made from old code. These survive the test suite (prose isn't tested) and mislead the next reader — often you, in a month. After a change, sweep for what it broke in words and artifacts, not just in code.

## Sweep after a change
- **Docstrings & signatures** → the function's own docstring still describes its real arguments, return, units, and side effects.
- **Comments near the change** → no comment now describes code that no longer exists or a formula you replaced.
- **Docs & README** → claims, examples, and quickstarts the new behavior contradicts — `grep` the old name/flag/value across the docs.
- **Figures & outputs** → cached plots/tables/notebooks generated by the old code are regenerated or marked stale (pairs with `artifact-first-reproducibility`).
- **Cross-references** → call sites and sibling modules that assumed the old contract.

## Anti-patterns
- "The tests pass, so it's done" — tests don't read docstrings; stale prose passes every test.
- Renaming a symbol and leaving its docstring, comments, and the README describing the old name.
- Changing units or defaults without grepping for every place the old value/assumption is asserted in words.
- Leaving committed figures generated by code that no longer produces them.

## Hard vs adaptable
- **Hard rule:** a change is not complete until the documentation/comments/artifacts it falsified are fixed or flagged. Confidently-wrong docs are worse than none.
- **Adaptable:** how wide to sweep — scale to blast radius (a private helper vs. a public API or a units change). A quick `grep` for the old name/value is the floor.

This is the *prose-and-artifact* half of finishing a change; removing the superseded *code* cleanly is `correct-cutover`.

## Related
- `correct-cutover` — removes the old code; this removes the old *claims about* the code.
- `decision-log-and-commits` — records why the change happened; this keeps the surrounding text true to it.
- `artifact-first-reproducibility` — figures/outputs falsified by a change must regenerate.
