Reference implementations are valuable scientific oracles, but they are not automatically reusable source material. Keep paper facts, runtime behavior, architecture inspiration, and copied implementation separate.

## Categories

| Category | Meaning | Typical status |
|---|---|---|
| Paper fact | Equation, table, definition, or stated algorithm in the paper | Reusable with citation |
| Runtime oracle | Run the reference code and compare outputs | Usually allowed for validation; cite and document version |
| Black-box value check | Use numerical outputs as regression/parity targets | Usually allowed; do not copy internals |
| Architecture pattern | High-level module boundary or phase decomposition | Usually allowed when independently expressed |
| Line translation | Porting code structure/expression line-by-line | High risk; avoid across incompatible licenses |
| Copied coefficients from code only | Values not present in paper/docs | Treat as code-derived; license and provenance review required |
| Redistribution | Shipping reference code, tables, or generated artifacts | Review license and data terms before including |

## Firewall Workflow

1. **Identify the reference.** Record repo, commit/tag, license, and role in the work.
2. **Classify each use.** Paper source, oracle, black-box target, architecture inspiration, or implementation reuse.
3. **Prefer paper/digest authority.** Exact equations and coefficients should come from verified papers or permitted data releases, not from reading source lines.
4. **For incompatible licenses, keep a clean room.** One pass can describe behavior and test cases; implementation should be independently derived from papers/specs.
5. **Log decisions.** Record what was used, what was not used, and which files/tests prove the boundary.

## Red Flags

- "This function is GPL, but I rewrote it in Python" without independent derivation.
- Coefficients copied from source comments when the paper table has not been checked.
- Reference-code special cases copied before understanding their physical regime.
- Test fixtures generated by reference code without version, input, compiler/options, or license note.
